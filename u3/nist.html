DAW 1º · NIST SP 800-145 (traducción técnica)

The NIST Definition of Cloud Computing (NIST SP 800-145)

Traducción y adaptación didáctica para 1º de DAW (desarrollo web y trabajo colaborativo). Basado en el documento oficial de NIST. :contentReference[oaicite:1]{index=1}

1) Definición de computación en la nube (traducción técnica)

Definición (NIST, traducción): La computación en la nube es un modelo que permite el acceso ubicuo, cómodo y bajo demanda, a través de la red, a un conjunto compartido de recursos informáticos configurables (por ejemplo: redes, servidores, almacenamiento, aplicaciones y servicios), que pueden aprovisionarse y liberarse rápidamente con un esfuerzo mínimo de gestión o interacción con el proveedor.
Este modelo se compone de cinco características esenciales, tres modelos de servicio y cuatro modelos de despliegue. :contentReference[oaicite:2]{index=2}
En DAW 1º: esta definición explica por qué hoy “montar” una aplicación web puede ser cuestión de minutos: el proveedor automatiza el aprovisionamiento (servidores, escalado, almacenamiento, etc.), y el equipo se centra en el código y la configuración.

2) Cinco características esenciales (NIST) — traducción

1. Autoservicio bajo demanda (On-demand self-service)
El consumidor puede aprovisionar capacidades (p. ej., tiempo de servidor o almacenamiento) de forma automática, cuando las necesita, sin intervención humana del proveedor. :contentReference[oaicite:3]{index=3}
2. Acceso amplio por red (Broad network access)
Las capacidades están disponibles en red y se accede a ellas con mecanismos estándar, facilitando su uso desde distintos dispositivos (móvil, tablet, portátil o PC). :contentReference[oaicite:4]{index=4}
3. Agrupación de recursos (Resource pooling)
Los recursos del proveedor se comparten para atender a múltiples consumidores mediante un modelo multi-inquilino (multi-tenant), asignando y reasignando recursos físicos/virtuales según demanda. Existe independencia de ubicación: el cliente no conoce ni controla la localización exacta, aunque puede seleccionar un nivel alto (país/region/CPD). :contentReference[oaicite:5]{index=5}
4. Elasticidad rápida (Rapid elasticity)
Las capacidades se aprovisionan y liberan de forma elástica (a veces automáticamente) para escalar hacia arriba o abajo según la demanda; para el consumidor puede “parecer” ilimitado. :contentReference[oaicite:6]{index=6}
5. Servicio medido (Measured service)
Los sistemas cloud controlan y optimizan el uso mediante medición (metering) según el tipo de servicio (almacenamiento, CPU, ancho de banda, usuarios activos), monitorizando y reportando consumo. Esto aporta transparencia para proveedor y consumidor. :contentReference[oaicite:7]{index=7}

3) Modelos de servicio (IaaS / PaaS / SaaS) — traducción + DAW

SaaS Software as a Service
Definición (traducción NIST)
Capacidad ofrecida al consumidor: usar las aplicaciones del proveedor que se ejecutan sobre infraestructura cloud. Accesibles desde distintos dispositivos mediante un cliente ligero (p. ej., navegador web) o una interfaz de programa. El consumidor no gestiona ni controla la infraestructura subyacente (red, servidores, SO, almacenamiento) ni las capacidades internas de la aplicación, salvo configuraciones limitadas específicas del usuario. :contentReference[oaicite:8]{index=8}
Ejemplos y relación con DAW 1º
  • GitHub: repos, issues, PRs → trabajo colaborativo y trazabilidad del proyecto web.
  • GitHub Actions: CI/CD como servicio → automatizar tests y despliegues.
PaaS Platform as a Service
Definición (traducción NIST)
Capacidad ofrecida al consumidor: desplegar en la infraestructura cloud aplicaciones creadas o adquiridas por el consumidor, usando lenguajes, bibliotecas, servicios y herramientas soportadas por el proveedor. El consumidor no gestiona ni controla la infraestructura subyacente (red, servidores, SO o almacenamiento), pero sí controla las aplicaciones desplegadas y, en ocasiones, la configuración del entorno de alojamiento. :contentReference[oaicite:9]{index=9}
Ejemplos y relación con DAW 1º
  • Vercel: despliegue de frontends (Next/React) con previews por PR.
  • Azure App Service: publicar una API web sin administrar el SO.
IaaS Infrastructure as a Service
Definición (traducción NIST)
Capacidad ofrecida al consumidor: aprovisionar procesamiento, almacenamiento, redes y otros recursos fundamentales, sobre los cuales el consumidor puede desplegar y ejecutar software arbitrario, incluyendo sistemas operativos y aplicaciones. El consumidor no gestiona ni controla la infraestructura cloud subyacente, pero sí controla el SO, el almacenamiento y las aplicaciones desplegadas, y posiblemente un control limitado de ciertos componentes de red (p. ej., cortafuegos del host). :contentReference[oaicite:10]{index=10}
Ejemplos y relación con DAW 1º
  • Amazon EC2 / Azure VMs: montar un servidor web real (Nginx/Apache), HTTPS, logs y despliegue manual.
  • Muy útil para entender “qué hay debajo” de PaaS (operación, parches, seguridad).
Clave DAW: SaaS = “uso una aplicación”; PaaS = “subo mi app y configuro”; IaaS = “administro el servidor + despliego”.

4) Modelos de despliegue — traducción

Nube privada (Private cloud)
Infraestructura cloud para uso exclusivo de una organización (con múltiples consumidores internos). Puede ser gestionada por la organización, por un tercero o combinaciones; puede estar dentro o fuera de las instalaciones. :contentReference[oaicite:11]{index=11}
Nube comunitaria (Community cloud)
Infraestructura cloud para uso exclusivo de una comunidad específica de organizaciones con preocupaciones compartidas (misión, requisitos de seguridad, políticas o cumplimiento). :contentReference[oaicite:12]{index=12}
Nube pública (Public cloud)
Infraestructura cloud disponible para el uso abierto del público general. Puede ser propiedad/operada por empresa, academia o administración pública; está en instalaciones del proveedor. :contentReference[oaicite:13]{index=13}
Nube híbrida (Hybrid cloud)
Composición de dos o más infraestructuras cloud distintas (privada, comunitaria o pública) que permanecen como entidades únicas, pero se integran con tecnología estandarizada o propietaria que permite portabilidad de datos y aplicaciones (p. ej., balanceo entre nubes). :contentReference[oaicite:14]{index=14}

Integración con Raspberry Pi, n8n e Inteligencia Artificial (contexto DAW 1º)

Raspberry Pi como laboratorio Cloud/Edge
Uso principal
Simular nube privada (on-prem) y/o edge/fog para servicios web reales (API + BBDD + reverse proxy) con coste casi cero.
Ejemplos DAW 1º
  • IaaS “local”: Docker + Nginx + app Node/PHP + PostgreSQL en la Pi.
  • Edge: cachear contenido, servir estáticos localmente, o procesar eventos antes de enviarlos a la nube.
  • Fog (gateway): la Pi como “pasarela” que agrega datos (sensores/logs) y los manda a cloud.
Clave didáctica: comparar el “control total” (Pi/IaaS local) vs rapidez (PaaS) vs comodidad (SaaS).
n8n (automatización e integraciones)
Qué es
Herramienta de automatización tipo “workflows” (webhooks, APIs, tareas programadas) para integrar servicios.
Clasificación (según despliegue)
  • SaaS: si se usa un servicio gestionado en la nube.
  • IaaS / On-prem: si se auto-hospeda en una VM o en Raspberry Pi.
Ejemplos DAW 1º
  • Webhook “nuevo PR” → crear tarjeta en tablero → avisar a chat → desplegar preview.
  • Webhook “nuevo alumno/entrega” → generar informe → guardar en drive → notificar.
  • Programar backups/exportaciones de BBDD y subir a storage.
Inteligencia Artificial en cloud (para DAW)
Qué aporta
IA como servicio para acelerar tareas del ciclo de vida: documentación, pruebas, soporte, análisis de logs, generación de contenido y asistentes conversacionales.
Encaje cloud
  • SaaS: herramientas completas (chat, edición, diseño, soporte).
  • PaaS/API: integrar IA en tu app web (chatbot, clasificación, búsqueda semántica).
  • Edge: respuestas rápidas y filtrado antes de llegar a cloud (cuando aplique).
Ejemplos DAW 1º
  • Generar borrador de README y documentación API (OpenAPI) y revisarlo críticamente.
  • Proponer casos de prueba unitarios y de integración y validarlos con el proyecto real.
  • Asistente FAQ para la web del proyecto (RAG: docs → respuestas).
Idea potente para el módulo: “Arquitectura híbrida educativa”. Raspberry Pi (fog/edge + laboratorio) + servicios PaaS/SaaS (despliegue rápido y colaboración) + IA (productividad).

Prácticas evaluables (comparativa de servicios + propuesta de arquitectura)

Mínimo requerido (por equipo)
  • 3 servicios IaaS comparados (p. ej., AWS EC2, Azure VM, Google Compute, DigitalOcean, Linode…).
  • 5 servicios PaaS (p. ej., Vercel, Netlify, Railway, Render, Fly.io, Azure App Service…).
  • 4 servicios SaaS (p. ej., GitHub, GitLab, Jira, Slack, Notion, Figma…).
  • 2 servicios Edge (p. ej., Cloudflare Workers, AWS Lambda@Edge, Fastly Compute…).
  • 1 ejemplo Fog/Mist:
    • Opción A: servicio IoT/gateway.
    • Opción B: Raspberry Pi como gateway fog + explicación teórica y práctica.
Workflow GitHub (obligatorio)
  1. Crear Issue para cada servicio a analizar.
  2. Crear rama desde la Issue: feature/comparativa-vercel
  3. Documentar en /docs/comparativas/vercel.md
  4. Abrir Pull Request y solicitar revisión de otro compañero.
  5. Merge tras aprobación (y checklist completo).
Extra recomendado: automatizar con GitHub Actions (lint/tests) y, si procede, notificación con n8n.

Arquitectura sugerida (ejemplo)
  • Frontend en PaaS (deploy desde Git).
  • API en PaaS o en IaaS (según control requerido).
  • BBDD gestionada (PaaS/DBaaS) o self-hosted (IaaS/Pi).
  • Edge para estáticos/caché (CDN/Workers).
  • Fog con Raspberry Pi como gateway opcional.
  • n8n para automatizar eventos (PRs, despliegue, notificaciones).
  • IA para documentación, testing y soporte (con validación humana).
Criterios que deben quedar evidentes
  • a) niveles cloud identificados (IaaS/PaaS/SaaS).
  • b) funciones cloud aplicadas (procesamiento, intercambio, ejecución, almacenamiento).
  • c) edge explicado y justificado en el caso.
  • d) fog/mist definidos y ubicados (si se usan).
  • e) ventajas justificadas con ejemplos (protección, interoperabilidad, movilidad, cooperación).
